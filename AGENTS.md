# AGENTS.md

本仓库全程由 AI Agent 独立开发与维护，Agent在开发维护过程中需多思考，多自主解决遇到的问题。研发核心原则不以「快速迭代完工」为目标，而是保障每一次代码变更**逻辑清晰、长期可维护**。

## 工作原则

- 优先修复问题根本成因，杜绝仅屏蔽告警、治标不治本的临时规避方案。
- 开发过程中遇显性错误、异常缺陷，应同步联动修复，而非临时绕过搁置。
- 每项开发任务完成后，必须执行对应模块测试，杜绝新增隐性问题与功能退化。
- 若修改脚本、路由规则、环境变量、默认端口、接口契约等核心配置，需同步联动更新 `README`、`.env.example`、相关文档、配套说明及相关测试用例。
- 项目如果缺少常见的代码质量 & 规范工具 & 配置（如 ESLint、Prettier、GitHub Actions CI、Dockerfile 优化等），Agent 需要主动补齐并且完善，提升项目质量与可维护性。
- 项目如果缺少测试工具与测试用例，Agent 需要主动补齐完善，覆盖核心功能与边界场景，提升项目质量与可维护性。
- 控制单文件代码体量：常规文件行数严控在 400 行以内，上限尽量不超过 600 行，便于 Agent 阅读与长期维护；少数情况确实需超限时，需要有合理理由。

## 提交与推送（Commit & Push）

一个批次/任务修改完成后Agent **可以自主决定**是否执行 `git commit` 与 `git push`，无需征求确认，前提是 Agent 已经自我验证了改动。
